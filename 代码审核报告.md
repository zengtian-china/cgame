### Bug 1【严重】读取存档未判空，空指针解引用
- **位置**：[main.c:111-112](file:///c:/Users/zengt/Desktop/game_copy/v1/main.c#L111-L112)

- **问题描述**：

  `read_save_main()` 在用户输入无效或文件读取失败时会返回 `NULL`（见 save.c:339、341）。`enterGame` 内部直接调用 `showUser(user)`、`getItemsList(user)` 等解引用 user，会导致程序崩溃。

### Bug 2【严重】scanf 未检查返回值，输入非法字符会死循环
- **位置**：[main.c:31](file:///c:/Users/zengt/Desktop/game_copy/v1/main.c#L31)、[main.c:42](file:///c:/Users/zengt/Desktop/game_copy/v1/main.c#L42)、[main.c:92](file:///c:/Users/zengt/Desktop/game_copy/v1/main.c#L92)

## 三、item.c

### Bug 1【严重】find_items 找不到物品时返回未初始化的内存
- **位置**：[item.c:82-94](file:///c:/Users/zengt/Desktop/game_copy/v1/item.c#L82-L94)
- **函数**：`find_items`
- **问题描述**：
  ```c
  items *item = newitems();   // newitems 仅 malloc，未初始化
  for(...) {
      if(list->array[i].id == id) { ... break; }
  }
  return item;   // 未找到时返回内容为垃圾值的 item
  ```
  调用方 battle.c 第 16 行 `if (it == NULL)` 永远不为真（除非 malloc 失败），后续 `sscanf(it->effect, ...)` 会读到乱码。
- **修改意见**：
  - 方案一：`newitems` 内部用 `calloc` 或 `memset` 清零，未找到时返回 NULL：
    ```c
  items *item = newitems();
  int found = 0;
  for(...) { if(...) { ...; found=1; break; } }
  if (!found) { free(item); return NULL; }
  return item;
  ```
  - 方案二：直接返回 `&list->array[i]` 指针，但要注意生命周期。

### Bug 2【严重】createItemsList 未检查 fopen 返回值

### Bug 3【严重】sort_item 合并相同物品时直接覆盖下一格
- **位置**：[item.c:184-189](file:///c:/Users/zengt/Desktop/game_copy/v1/item.c#L184-L189)
- **函数**：`sort_item`
- **问题描述**：
  ```c
  if(user->invertory[i].item_count + user->invertory[j].item_count > 99){
      user->invertory[i].item_count = 99;
      user->invertory[i+1].item_count = tmp_x+tmp_y-99;   // 直接覆盖 i+1 格
      break;
  }
  ```
  1) 未检查 `i+1` 是否越界（i 可能等于 size-1）。
  2) 如果 `invertory[i+1]` 已经存在别的物品，会被直接覆盖，导致物品丢失。
  3) 该格原本的物品 ID 未设置，可能为 0 或其它值，造成数据混乱。
- **修改意见**：合并时应该把溢出的数量放到背包末尾的新格子（`invertory[invertory_size]`），并增加 `invertory_size`，同时检查是否超过 MAX_ITEMS。

### Bug 4【严重】sort_item 中 invertory_size 重复递减
- **位置**：[item.c:206-210](file:///c:/Users/zengt/Desktop/game_copy/v1/item.c#L206-L210)
- **函数**：`sort_item`
- **问题描述**：
  ```c
  for (int i = slow; i < size; i++) {
      user->invertory[i].item_id = 0;
      user->invertory[i].item_count = 0;
      user->invertory_size -= 1;     // 每次循环都减1
  ```

### Bug 7【中等】sort_item 排序后合并逻辑可能遗漏
- **位置**：[item.c:173-198](file:///c:/Users/zengt/Desktop/game_copy/v1/item.c#L173-L198)
- **函数**：`sort_item`
- **问题描述**：合并循环中 `break` 后，i+1 位置的物品未被处理，但下一轮 i++ 会跳过它。例如背包有 [1,1,1,2]，i=0 合并 i=1，break；i=1 位置变成 item_id=1, count=0；i=1 时跳过；i=2 时找 j=3，id 不同不合并。最终 [1,1,1,2] 被压成 [1,2] 但中间的 1 数量丢失。

### Bug 8【轻微】find_items 未找到时返回的 item 在调用方未释放

## 六、player.c

### Bug 1【严重】levelUpLogic 死循环风险
- **位置**：[player.c:117-157](file:///c:/Users/zengt/Desktop/game_copy/v1/player.c#L117-L157)
- **函数**：`levelUpLogic`
- **问题描述**：
  ```c
  int max_exp = 0;
  if (user->level < 10) {
      max_exp = user->level * 100;
      while (max_exp <= user->exp) {
          user->exp = user->exp - max_exp;
          levelUp(user);    // level 增加，但 max_exp 没重新计算
      }
  }
  ```
  `max_exp` 在 while 外面计算一次，循环内 `levelUp` 让 level 增加，但 max_exp 没更新。如果 level 从 9 升到 10，max_exp 还是 900，但实际下一级需要 10*200+10*10*2=2200。如果 exp 很大，可能死循环（如果 max_exp 不变且 exp 一直大于 max_exp）。实际上 levelUp 后 level 变化，但 if/else if 分支不会再进入（因为已选择分支），所以 max_exp 不变，可能死循环。
  
  例如：level=9，exp=10000，max_exp=900。每次循环 exp -= 900，level+1。level 到 10 后，仍用 max_exp=900，exp 还很大，继续升级到 11、12……直到 level>=10 后才停止吗？不，if 分支已经选定，max_exp 不变，会一直升级到 exp < 900。但是 levelUp 内部 level>=10 后会走 else if 分支，属性成长不同，但 max_exp 仍然是 900，会疯狂升级，导致 level 飙升到异常值。
- **修改意见**：把 max_exp 计算放到 while 循环内，每次升级后重新计算：
  ```c
  while (1) {
      int max_exp;
      if (user->level < 10) max_exp = user->level * 100;
      else if (user->level < 30) max_exp = user->level*200 + user->level*user->level*2;
      else if (user->level < 40) max_exp = user->level*400 + user->level*user->level*5;
      else if (user->level < 50) max_exp = user->level*100 + user->level*user->level*10;
      else break;
      if (max_exp > user->exp) break;
      user->exp -= max_exp;
      levelUp(user);
  }
  ```

## 九、跨文件 / 全局问题

### Bug 1【严重】scanf 全局未检查返回值
- **位置**：几乎所有 .c 文件
- **问题描述**：项目中所有 `scanf("%d", &x)` 都未检查返回值，输入非数字会导致变量保持原值、缓冲区未清空、死循环。这是项目最普遍的 bug。
- **修改意见**：封装一个工具函数：
  ```c
  int read_int(int *out) {
      if (scanf("%d", out) != 1) {
          int c; while ((c = getchar()) != '\n' && c != EOF);
          return 0;
      }
      return 1;
  }
  ```
  全项目替换。

### Bug 2【严重】map.h 中重复 include 自身

### Bug 3【中等】MAX_ITEMS 在多处重复定义

### Bug 4【中等】battle.h 中前置声明 User 和 monster 但已有完整定义
